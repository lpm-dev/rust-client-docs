```bash
lpm doctor [--fix] [-y]
lpm health
```

Two complementary diagnostic commands:

- **`lpm doctor`** ŌĆö checks the **project + environment**: Node version detection, `lpm install` state, sandbox availability, certificate trust, store integrity, and a handful of others. The big "is everything OK here?" command.
- **`lpm health`** ŌĆö checks the **registry**: connectivity, latency, the registry's own health endpoint. The "is the service up?" command.

## Examples

```bash
lpm doctor                       # report ŌĆö read-only
lpm doctor --fix                 # auto-fix what it can, prompt where unclear
lpm doctor --fix -y              # auto-fix, no prompts (CI-friendly)
lpm doctor --json                # structured for tooling

lpm health                       # registry connectivity check
lpm health --json
```

## What `lpm doctor` checks

The check set is intentionally broad ŌĆö every signal that "this project is in a known-good state to develop in." Highlights:

- Node version detected (`.nvmrc` / `lpm.json > runtime.node`) and installed
- `node_modules/` matches the lockfile (no drift since the last `lpm install`)
- Sandbox availability (Seatbelt on macOS, landlock on Linux)
- Local CA exists and is trusted (for `lpm dev --https`)
- Store integrity (no missing or corrupted entries)
- TypeScript reachability per `tsconfig.json` (workspace-aware): does `tsc` resolve via project-local `node_modules/.bin`, only via system `PATH`, or not at all? See [`lpm check`](/docs/dev/check) for the readiness contract.
- Auth tokens present and not expired

`--json` returns the full per-check result set; `--verbose` adds remediation hints inline.

## Machine-readable output (`--json`)

`lpm doctor --json` is the structured surface every CI pipeline, MCP server, or scripting tool should consume ŌĆö it's a contract, not a debug aid. The shape is stable; consumers should match on `code`, never on the human-readable `check` or `detail` strings.

```json
{
  "success": true,
  "no_failures": false,
  "clean": false,
  "has_warnings": true,
  "passed": 12,
  "failed": 1,
  "warnings": 2,
  "fixes_applied": [],
  "checks": [
    {
      "code": "registry_reachable",
      "check": "Registry",
      "passed": true,
      "severity": "pass",
      "detail": "https://lpm.dev"
    },
    {
      "code": "auth_invalid",
      "check": "Authentication",
      "passed": false,
      "severity": "fail",
      "detail": "token exists but invalid ŌĆö run: lpm login"
    }
  ]
}
```

Per-check fields:

| Field | Type | Notes |
| --- | --- | --- |
| `code` | `string` | Stable snake_case identifier. Examples: `registry_unreachable`, `node_modules_missing`, `pnpm_overrides_drift`. **Match on this.** |
| `check` | `string` | Human-readable label. Wording can change. |
| `passed` | `bool` | `true` for `pass` and `warn`; `false` for `fail`. |
| `severity` | `string` | One of `pass` \| `fail` \| `warn`. |
| `detail` | `string` | Human-readable detail with remediation. May be empty. |

`severity = "warn"` carries `passed = true` so that warning-only runs don't trip strict CI gates that key off `no_failures`. Use `has_warnings` (top-level) or filter `checks[].severity == "warn"` if you want to surface warnings explicitly.

Exit code: `0` when no `severity = "fail"` checks fired (warnings are tolerated), `1` otherwise.

## Auto-fix mode

```bash
lpm doctor --fix
```

Walks the failing checks and asks before each remediation. Common fixes include:

- Install the missing Node version
- Run `lpm install` to reconcile `node_modules/`
- Run `lpm fmt` to reconcile formatting
- Install the local CA into the OS trust store

Pass `-y` (which implies `--fix`) to skip every prompt ŌĆö useful in CI bootstrap scripts.

## What `lpm health` checks

```bash
lpm health
```

Hits the registry's health endpoint, measures round-trip latency, and reports up/down + p50/p99. Use as a CI smoke test or when investigating slow installs.

`lpm health` does not check your project ŌĆö only the registry. For project + environment checks, use `lpm doctor`.

## Flags

### `lpm doctor`

| Flag | Effect |
| --- | --- |
| `--fix` | Auto-fix issues with prompts |
| `-y`, `--yes` | Skip prompts (implies `--fix`) |

### `lpm health`

No specific flags besides the [global flags](/docs/commands#global-flags) ŌĆö `--json` and `--verbose` are useful.
