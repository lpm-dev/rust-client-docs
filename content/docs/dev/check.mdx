title: lpm check
description: "Type-check the project with tsc --noEmit."
---

```bash
lpm check [-- args...]
```

Runs `tsc --noEmit` against the project. Forwards any trailing arguments to `tsc`.

Unlike [`lpm lint`](/docs/dev/lint) and [`lpm fmt`](/docs/dev/fmt), `lpm check` is **not** lazy-downloaded â€” TypeScript must be installed in the project (or globally available on `PATH`). LPM resolves `tsc` from `node_modules/.bin` via the same PATH-injection it uses for `lpm run`.

## Examples

```bash
lpm check                          # type-check the whole project
lpm check -- --pretty              # forward tsc flags
lpm check -- -p tsconfig.test.json # type-check a different tsconfig
lpm check --all                    # workspace: every member
lpm check --filter web             # workspace: one member
lpm check --filter './apps/*'      # workspace: path glob
lpm check --affected               # workspace: only affected members
```

## Requires TypeScript in the project

```bash
lpm install -D typescript
```

`lpm check` resolves `tsc` from `node_modules/.bin` first and falls back to the system `PATH`. The project-local install is the right answer â€” editor IntelliSense (`tsserver` from `node_modules/typescript`) and CI run the same version, and your `tsconfig.json` references resolve against the same `@types/*` packages.

### Preflight

Before spawning `tsc`, `lpm check` verifies setup and surfaces specific errors:

| Condition | Error |
| --- | --- |
| `tsconfig.json` is missing | `no tsconfig.json found in <dir>. Add one (or pass -p <path>) to use a different one.` |
| `typescript` declared in `package.json` but not installed | `typescript declared in package.json but not installed. Run: lpm install` |
| `typescript` not declared and no `tsc` on `PATH` | `typescript not installed. Run: lpm install -D typescript` |

The preflight is **argument-aware**: when you pass `-p <path>`, `--project <path>`, or a positional file argument, the tsconfig check is skipped and `tsc` decides â€” `lpm check -- -p tsconfig.test.json` and `lpm check src/foo.ts` both work without a root `tsconfig.json`.

### `lpm doctor` reports the same signal

`lpm doctor` emits one TypeScript check per `tsconfig.json` in the workspace, with stable codes:

| Code | Severity | Meaning |
| --- | --- | --- |
| `typescript_healthy` | pass | `tsc` resolves through the project-local `node_modules/.bin` chain. Editor + CI parity holds. |
| `typescript_missing_for_tsconfig` | warn | `tsc` runs only via the system `PATH`. The project lacks a local install â€” editor and CI may use a different version. Run `lpm install -D typescript` to converge. |
| `typescript_unavailable` | fail | `tsc` cannot run at all. Run `lpm install` if `typescript` is already declared, or `lpm install -D typescript` to add it. |

Doctor performs only the cheap reachability + dep-declaration check; the type-check itself belongs to `lpm check`.

## Workspaces

```bash
lpm check --all                              # every member
lpm check --filter web                       # exact name
lpm check --filter '@scope/*'                # glob
lpm check --filter './apps/*'                # path glob
lpm check --affected --base develop          # affected vs a non-main branch
lpm check --filter web --fail-if-no-match    # exit non-zero on typo'd filter
```

Members run in topological levels, with packages inside each level executing in parallel up to the available CPU count.

Filter grammar is documented in [Workspaces](/docs/packages/workspaces#filter-grammar).

`--all` and `--affected` are mutually exclusive; `--filter` composes with `--affected` (the affected set is unioned with the filter result).

## Flags

| Flag | Effect |
| --- | --- |
| `--all` | Run in every workspace member |
| `--filter <expr>` | Select workspace members by the [filter grammar](/docs/packages/workspaces#filter-grammar) (repeatable; entries union) |
| `--affected` | Run only in members affected by changes vs `--base` |
| `--base <REF>` | Git base ref for `--affected` (default: `main`) |
| `--fail-if-no-match` | Exit non-zero if no member matches the filter set (recommended in CI) |

Anything after `--` (or trailing) is forwarded to `tsc`. `--noEmit` is always passed.

Plus the [global flags](/docs/commands#global-flags).

## `--json` in workspace mode

Single-package mode (`lpm check`) preserves tsc's stdout â€” LPM does not wrap it. Workspace mode emits a single LPM envelope on stdout; per-member stdout/stderr is captured and surfaced inside the envelope only on failure. See [`lpm lint`](/docs/dev/lint#json-in-workspace-mode) for the exact envelope shape.
